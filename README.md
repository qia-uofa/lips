# LIPS — LLM-driven Iterative Project Synthesizer

LIPS is a pipeline framework that uses large language models to iteratively generate, refine, and verify software projects. You define stages and build scripts using plain Markdown files; LIPS handles the rest.

---

## How It Works

LIPS organizes work into **pipelines** and **stages**. Each stage has a `repo/` directory (the working files) and a `build/` directory (the prompt instructions). When you run a build, LIPS assembles a message from the source stage's repo, optionally the target stage's repo, and the prompt file, then asks an LLM to generate or update files.

```
workspace/
└── my-pipeline/
    ├── .env
    ├── api-config.json
    ├── stage-a/
    │   ├── build/
    │   │   └── compile.md        ← prompt for this build mode
    │   └── repo/
    │       └── ...               ← source files
    └── stage-b/
        ├── build/
        │   └── compile.md
        └── repo/
            └── ...               ← generated/updated files
```

## Getting Started

### 1. Install

```bash
python -m pip install git+https://github.com/mandelbroetchen/lips.git
```

Or clone and install locally:

```bash
git clone https://github.com/mandelbroetchen/lips
cd lips
python -m pip install -e .
```

### 2. Create a pipeline

Run the interactive wizard from any directory:

```bash
lips create
```

This prompts you for a workspace root, pipeline name, LLM provider and model, API key, and the names of your stages. It writes the directory structure, an `api-config.json`, and a `.env` file.

### 3. Configure

LIPS reads two config files from your **working directory** when you invoke it — so you can keep them anywhere, as long as you `cd` there first (or pass `--api-config` to point to them explicitly).

**`.env`**
```
MISTRAL_API_KEY=sk-...
```

**`api-config.json`**
```json
{
    "generate":{
        "model": "mistral/mistral-large-latest",
        "max_tokens": 20000,
        "temperature": 0,
        "timeout": 1200
    },

    "api_var": "MISTRAL_API_KEY"
}
```

Any model string supported by [LiteLLM](https://docs.litellm.ai/docs/providers) works here.

### 4. Build

```bash
lips build <path/to/stage>
```

This runs the default `compile` mode, reading `build/compile.md` from the given stage. The `TARGET` variable in that file determines which stage receives the generated output.

## Build scripts

Every `.md` file you place in a stage's `build/` directory becomes a build mode. The file name (without extension) is the script name.

```bash
lips build my-script path/to/stage
```

This reads `path/to/stage/build/my-mode.md` as the prompt. You can define as many scripts as you like — `compile.md`, `verify.md`, `patch.md`, `refactor.md`, etc.

Inside each prompt file, a small `env` block tells LIPS where the output goes:

````markdown
```env
TARGET=stage-b
```

Your prompt instructions here...
````

## Example Pipeline: `python-project`

A four-stage workflow for building a Python project through iterative LLM-guided refinement: from initial concept through specification, implementation, and verification.

### Directory Structure

```
workspace/
└── python-project/
    ├── 1-prompt/          ← Raw idea (human-authored seed)
    ├── 2-concept/         ← Structured specification (LLM-generated)
    ├── 3-specs/           ← Detailed design (LLM-generated from concept)
    ├── 4-target/          ← Working implementation (LLM-generated from specs)
    └── 5-docs/            ← Documentation and verification logs
```

### Workflow

```
1-prompt ──compile──▶ 2-concept ──compile──▶ 3-specs ──compile──▶ 4-target
                                                                       │
        ┌──────────────────────────────────────────────────────────────┘
        │
      verify  →  writes *.verify.md (verification findings)
        │
      redeem   →  reads *.verify.md, fixes issues, clears markers
```

### Stage Descriptions

#### **1-prompt** → Start Here
- **Purpose**: Capture the project idea in natural language
- **Author**: You (manual input)
- **Content**: Problem description, goals, constraints, examples

#### **2-concept** → Generated Specification
- **Purpose**: Transform the raw idea into a structured specification
- **Generated by**: `build prompt compile`
- **Instruction file**: `1-prompt/build/compile.md`
- **Output**: High-level design, module breakdown, API outlines in `2-concept/repo/`

#### **3-specs** → Detailed Design
- **Purpose**: Expand the specification with implementation details
- **Generated by**: `build concept compile`
- **Instruction file**: `2-concept/build/compile.md`
- **Output**: Function signatures, data structures, algorithm outlines in `3-specs/repo/`

#### **4-target** → Working Code
- **Purpose**: Produce the final, runnable implementation
- **Generated by**: `build specs compile`
- **Instruction file**: `3-specs/build/compile.md`
- **Output**: Complete, tested source code in `4-target/repo/`

#### **5-docs** → Verification & Refinement
- **Purpose**: Track issues and improvements across cycles
- **Verify mode**: `build target verify` → writes `*.verify.md` files with findings
- **Redeem mode**: `build target redeem` → fixes issues based on `*.verify.md`, then clears markers

### Running a Complete Cycle

```bash
cd workspace/python-project

# Generate each stage in sequence
python3 -m lips build 1-prompt
python3 -m lips build 2-concept
python3 -m lips build 3-specs
python3 -m lips build 4-target

# Verify the implementation
python3 -m lips build verify 4-target

# Fix identified issues (repeat until clean)
python3 -m lips build redeem 4-target
python3 -m lips build verify 4-target  # Check again
```

### Iteration Loop

1. **Review** → Check `*.verify.md` files in `4-target/repo/` for issues
2. **Decide** → Determine if findings are valid and worth fixing
3. **Redeem** → Run `build redeem 4-target` to auto-fix problems
4. **Re-verify** → Run `build verify 4-target` again
5. **Repeat** → Continue until no issues remain

The `*.verify.md` files are `.gitignore`d, so they won't clutter your actual codebase.

### Key Points

- **Incremental**: Each stage builds on the previous one
- **Reversible**: Edit any stage's source and recompile downstream
- **Self-correcting**: Verification loop catches bugs before final output
- **Documented**: Specification stages create a clear paper trail

## CLI Reference

```
python3 -m lips build [mode] <stage-path> [--api-config <path>]
python3 -m lips purge <stage-path>
python3 -m lips purge --pipeline <pipeline-path>
python3 -m lips create
```

| Command | Description |
|---|---|
| `build [mode] <stage>` | Run a build script against a stage. Defaults to `compile`. |
| `purge <stage>` | Delete all generated files in a stage's `repo/` (preserves `.gitignore`). |
| `purge --pipeline <dir>` | Purge all stages in a pipeline. |
| `create` | Interactive wizard to scaffold a new pipeline. |

The `--api-config` flag defaults to `api-config.json` in your current working directory. The `.env` file is also loaded from the working directory. This means you can store both files at the pipeline root and simply `cd` there before invoking LIPS — or keep a shared config elsewhere and pass the path explicitly.

## Supported Providers

LIPS uses [LiteLLM](https://docs.litellm.ai) under the hood, so any provider it supports works out of the box. The `lips create` wizard offers shortcuts for:

| Provider | Example model string |
|---|---|
| Mistral | `mistral/mistral-large-latest` |
| OpenAI | `openai/gpt-4o` |
| Anthropic | `anthropic/claude-opus-4-6` |
| Google | `google/gemini-2.0-flash` |

For any other provider, enter the model string manually when prompted.

## File Format

Generated files are returned by the LLM in a simple XML envelope:

```xml
<file path="/absolute/path/to/file.py">
# file contents here
</file>
```

LIPS parses these blocks and writes each file to disk. Your prompt files can include an `ignore` block to exclude patterns from the repo snapshot sent to the model:

````markdown
```ignore
*.verify.md
__pycache__/
```
````
